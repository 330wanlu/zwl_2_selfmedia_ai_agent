# 美妆自媒体运营 AI Agent — 优化方向与未来规划

> 版本：v1.0　|　日期：2026-07-12　|　基于 MVP（阶段 0～4）现状整理

---

## 一、当前 MVP 已具备的能力（基线）

| 模块 | 已实现 |
| --- | --- |
| 工作流 | LangGraph 五节点流水线 + 三个人工断点（选题 / 文案 / 图片） |
| 持久化 | PostgreSQL 业务表 + LangGraph PostgresSaver checkpoint，服务重启可续跑 |
| 内容产出 | 小红书内容包（标题 / 正文 / 标签 / 配图），JSON 导出 + 图片 zip |
| 前端 | 任务列表、详情三决策区、内容包页，3 秒轮询 |
| 调试 | `llm_call_logs` 全量埋点、DEBUG 端点查 State / 历史 / 调用日志 |
| 边界 | 文案多轮修改、单张重绘、任务取消、并发任务隔离 |
| 自测 | 后端 `test_stage4.py`、前端 Playwright E2E |

## 二、当前需要优化的点

按优先级分为 **P0（影响日常使用）**、**P1（体验与稳定性）**、**P2（工程质量）**。

### P0 — 影响核心使用体验

| 序号 | 优化项 | 现状问题 | 建议方向 |
| --- | --- | --- | --- |
| 1 | **全流程耗时** | 单篇 E2E 约 3 分钟（不含人工审核），文本调用 30s+、生图约 20s/张，多轮修改/重绘成本线性上升 | 节点级并行（多张图并发调 Seedream）、Prompt 压缩减少 token、对「换一批选题」等操作加防抖/冷却 |
| 2 | **LLM 调用成本与额度** | 无预算上限、无重试退避策略，边界测试（revise5 + redraw3）易快速消耗额度 | 任务级调用次数/费用统计、超阈值告警、失败重试指数退避 |
| 3 | **图片存储** | 图片存本地 `data/images/`，无清理策略，长期运行磁盘膨胀 | 定期清理策略、按任务归档；生产环境接入对象存储（OSS/S3） |
| 4 | **错误恢复** | 任务 `failed` 后缺少「从断点重试」入口，只能重新建任务 | 基于 checkpoint 提供「恢复任务」API + 前端按钮 |
| 5 | **运营误操作防护** | 取消后不可恢复；选题/通过等关键操作缺少二次确认（部分有，可统一） | 关键决策二次确认、取消前弹窗说明后果 |

### P1 — 体验与产品化

| 序号 | 优化项 | 现状问题 | 建议方向 |
| --- | --- | --- | --- |
| 6 | **实时状态推送** | 前端 3 秒轮询，生成图片阶段用户感知延迟明显 | 升级为 SSE 或 WebSocket，详情页即时刷新 |
| 7 | **任务列表信息密度** | 列表仅展示方向与状态，缺少耗时、版本号、失败原因摘要 | 增加创建时间、耗时、当前版本、错误信息列 |
| 8 | **文案版本对比** | `content_versions` 已落库全部版本，前端只展示最新版 | 版本时间线 + diff 对比，支持回滚到历史版本 |
| 9 | **图片预览体验** | 九宫格预览基础可用，缺少放大、对比重绘前后 | Lightbox 大图、重绘前后并排对比 |
| 10 | **内容包页增强** | 一键复制与 zip 下载已有，缺少「复制标题」「复制标签」分项 | 分项复制按钮、发布 checklist（标题长度、标签数校验提示） |
| 11 | **Prompt 持续调优** | 阶段 4 已打磨一轮，不同品类（口红/防晒/底妆）质量仍有波动 | 建立品类 Prompt 模板、A/B 评测脚本常态化 |
| 12 | **图片中文排版** | Seedream 短中文（≤6 字）可用，长文/复杂排版不稳定 | 评估「底图生成 + Pillow 程序化排版」混合方案，按场景切换 |
| 13 | **图片风格一致性** | 多张图风格偶有不统一 | 分镜 Prompt 强化统一 style token；可选参考图/风格预设 |
| 14 | **挂起任务治理** | 联调后易残留 `waiting_*` 任务，列表显得「卡住」 | 列表筛选（进行中/挂起/已完成）、批量取消、超时自动提醒 |

### P2 — 工程质量与可运维性

| 序号 | 优化项 | 现状问题 | 建议方向 |
| --- | --- | --- | --- |
| 15 | **用户认证与权限** | MVP 无登录，单人本地使用 | JWT/OAuth2 + 角色（运营/管理员），API 鉴权中间件 |
| 16 | **部署与容器化** | 本机原生 PostgreSQL + 手动启停，无 Docker / CI | Docker Compose 一键起、GitLab CI 跑 lint + 冒烟测试 |
| 17 | **可观测性** | `llm_call_logs` 有，LangSmith 未开启 | 可选接入 LangSmith 全链路 trace；结构化日志 + 请求 ID |
| 18 | **单元测试覆盖** | 以脚本化 E2E 为主，缺少节点级单测 | 对 `json_utils`、路由函数、gate 逻辑补 pytest；Mock LLM |
| 19 | **API 文档与类型** | Swagger 已有，前后端类型未 codegen 同步 | OpenAPI → TypeScript 类型生成，减少手工维护 |
| 20 | **数据库迁移规范** | Alembic 已用，checkpoint 表单独初始化 | 文档化「业务迁移 vs checkpoint 初始化」流程；staging 环境验证 |
| 21 | **配置管理** | `.env` 本地配置，无分环境 | dev/staging/prod 配置分离，密钥走密钥管理服务 |
| 22 | **Windows 开发体验** | PowerShell GBK 乱码、ini 中文注释踩坑（已部分规避） | 开发文档统一 UTF-8 约定；可选 devcontainer |

## 三、未来值得添加的功能

按业务价值与实现复杂度分层。

### 近期（1～2 个迭代，在现有架构上扩展）

| 功能 | 说明 | 架构落点 |
| --- | --- | --- |
| **多平台内容包** | 同一任务导出抖音/快手/公众号适配格式（仍人工发布） | `prompts/platform/` + `publishers/` 各加适配器 |
| **任务模板 / 方向库** | 常用方向（如「敏感肌防晒」「平价口红」）一键创建 | 新表 `direction_templates` + 列表页快捷入口 |
| **内容品鉴看板** | 汇总可发率、平均耗时、标签/标题合规率 | 读 `publish_records` + `llm_call_logs` 聚合 API |
| **Prompt 后台可视化** | 运营可调 Prompt 而不改代码 | `prompts` 入库 + 管理页，带版本号与回滚 |
| **SSE 实时进度** | 替代轮询，展示当前节点与百分比 | `api/v1/events.py` + 前端 EventSource |
| **历史优质文案学习** | 从 marked published 的内容提炼风格偏好 | 检索增强或 fine-tune 前的数据管道 |

### 中期（需一定基建，2～4 周量级）

| 功能 | 说明 | 架构落点 |
| --- | --- | --- |
| **微信公众号 API 直发** | 官方有成熟发布 API，适合图文 | `publishers/wechat_mp.py` + 素材上传 |
| **抖音 H5 扫码半自动发布** | 官方合规「拉起发布」，用户 App 内确认 | `publishers/douyin_h5.py` |
| **定时/批量排产** | 一次录入多个方向，按队列生成 | APScheduler + 任务队列（Redis/Celery 可选） |
| **平台账号管理** | 存储各平台凭证（加密） | `platform_accounts` 表 + `core/security.py` |
| **对象存储** | 图片上云，支持 CDN | `core/storage.py` 抽象，本地/OSS 可切换 |
| **多模型路由** | 按节点或任务类型选不同文本/生图模型 | `llm/factory.py` 扩展 provider |
| **子图：仅重出图** | 已定稿文案，单独跑分镜+出图 | LangGraph Subgraph 拆分 |
| **团队协作** | 多运营、任务指派、审核权责 | 用户体系 + `tasks.assignee_id` |

### 远期（产品化 / 规模化）

| 功能 | 说明 |
| --- | --- |
| **数据回流分析** | 对接平台数据（阅读、点赞、转化），反哺选题与 Prompt |
| **记忆与偏好学习** | 长期记住品牌调性、禁用词、常用标签 |
| **视频脚本与短视频** | 图文流水线延伸为 15～60s 脚本 + 分镜 |
| **AIGC 合规检测** | 敏感词、广告法、平台违禁词自动扫描 |
| **RPA 自动发布（高风险）** | Playwright 操作创作者中心，需严格风控与人工兜底 |
| **AI 服务独立部署** | `agent/` + `llm/` 拆为微服务，FastAPI 经 RPC 调用 |
| **多租户 SaaS** | 按品牌/客户隔离数据与配额 |

## 五、与《项目方案设计》的差异说明

MVP 相对完整方案已**有意裁剪**以下能力，上文「未来功能」即对应回填路径：

| 完整方案有、MVP 暂无 | 回填方式 |
| --- | --- |
| 用户认证 / 权限 | `core/security.py` + API 依赖注入 |
| SSE 实时推送 | `api/v1/events.py` |
| 多平台发布 | `publishers/` + `prompts/platform/` 插件 |
| 定时批量发布 | 任务服务 + APScheduler |
| Docker 部署 | 生产期 docker-compose |
| 对象存储 | `core/storage.py` |
| 数据回流 / 记忆学习 | 独立分析模块 |

架构上已预留扩展点（工作流与 API 解耦、Prompt 集中管理、发布适配器模式），加功能时**主流程 LangGraph 图可保持不变**。

## 六、验收口径（优化时可对照）

阶段 4 已建立的「可发」自动化口径，后续优化应不降级：

- 标题：去 emoji 后约 8～22 字
- 正文：400～1500 字，含互动提问
- 标签：5～10 个
- 图片：≥3 张，短中文可读
- 边界：文案 5 轮修改、单张重绘 3 次、取消后决策 409、并发任务隔离

任何 Prompt 或流程改动，建议先跑 `test_stage4.py review` 再跑抽样 E2E。
