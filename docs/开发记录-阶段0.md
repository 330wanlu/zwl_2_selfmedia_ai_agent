## 5. 数据库迁移说明（为什么要迁移、流程是什么）

### 5.1 一句话

**建表迁移 = 用代码把「数据库应该长什么样」记录下来，并按步骤自动执行**，而不是每次手写 SQL，或靠程序启动时偷偷建表。

迁移的目的不是「搬家迁数据」，而是**可重复、可版本化、可演进地管理表结构**。

### 5.2 我们项目里建了什么表？

| 类别 | 表名 | 作用 |
|------|------|------|
| 业务表 | `tasks` | 任务（方向、状态、当前阶段、LangGraph thread_id） |
| 业务表 | `topics` | AI 生成的候选选题 |
| 业务表 | `content_versions` | 文案各版本（含人工反馈历史） |
| 业务表 | `images` | 分镜生成的图片 |
| 业务表 | `publish_records` | 小红书内容包 / 发布记录 |
| 业务表 | `llm_call_logs` | 每次 LLM/生图调用日志（调试用） |
| LangGraph | `checkpoints` 等 4 张 | 工作流状态持久化（interrupt 恢复用） |
| Alembic | `alembic_version` | 记录数据库已执行到哪个迁移版本 |

这些表 PostgreSQL 不会自动提供，**必须有人创建**。

### 5.3 三种建表方式对比

| 方式 | 做法 | 缺点 |
|------|------|------|
| A. 手动 SQL | 在 pgAdmin 里手写 `CREATE TABLE` | 换环境要重做；改表容易漏；无版本记录 |
| B. 程序启动建表 | `Base.metadata.create_all()` | 不好处理「表已存在、要加字段」；无迁移历史 |
| **C. Alembic 迁移（本项目采用）** | 迁移脚本进 Git，`upgrade` 按版本执行 | 可重复、可演进、团队环境一致 |

### 5.4 本项目实际执行的流程

```
① 在 Python 定义 ORM 模型
   app/models/task.py、topic.py、content.py ...
        ↓
② Alembic 对比「模型定义」与「当前数据库」，自动生成迁移脚本
   uv run alembic revision --autogenerate -m "init business tables"
        ↓
   生成：alembic/versions/26321e0bb705_init_business_tables.py
   （内含 CREATE TABLE tasks、topics ... 等操作）
        ↓
③ 执行迁移，在 PostgreSQL 里真正建表
   uv run alembic upgrade head
        ↓
   media_agent 库出现 6 张业务表 + alembic_version 记录当前版本
        ↓
④ LangGraph checkpoint 表单独初始化（不走 Alembic）
   uv run python scripts/setup_checkpointer.py
        ↓
   再出现 checkpoints、checkpoint_blobs 等 4 张表
```

阶段 0 对应操作顺序：

1. `scripts/create_db.py` → 创建空库 `media_agent`（只有库，还没有表）
2. `alembic revision --autogenerate` → 根据模型生成迁移脚本
3. `alembic upgrade head` → 执行业务表建表
4. `scripts/setup_checkpointer.py` → 初始化 LangGraph 专用表

### 5.5 类比理解

| 概念 | 类比 |
|------|------|
| ORM 模型（`app/models/*.py`） | 房子设计图纸 |
| 迁移脚本（`alembic/versions/*.py`） | 施工步骤说明书 |
| `alembic upgrade head` | 按说明书施工 |
| `alembic_version` 表 | 施工日志：目前已完工到第几步 |
| 以后的新迁移 | 装修改造说明书：在现有结构上增改 |

### 5.6 Alembic 迁移 vs checkpoint 初始化

| | Alembic 迁移 | `setup_checkpointer.py` |
|--|--------------|---------------------------|
| 管什么 | **我们自己的业务表** | LangGraph **官方规定**的 checkpoint 表 |
| 结构由谁定义 | `app/models/` | LangGraph 库内部 |
| 怎么建 | `alembic upgrade head` | `PostgresSaver.setup()` |

两套都必要：业务表存任务/选题/文案/图片；checkpoint 表存工作流跑到哪、interrupt 时 State 是什么。

### 5.7 以后还会用到吗？

会。例如阶段 1 要给 `tasks` 加字段 `error_message`：

```
① 改 app/models/task.py
② uv run alembic revision --autogenerate -m "add task error_message"
③ 检查生成的迁移脚本
④ uv run alembic upgrade head
```

数据库平滑升级，不用删库重建，已有数据保留。

### 5.8 常用命令速查

```powershell
# 在 backend 目录下执行

# 根据模型变更自动生成迁移脚本（改 models 后用）
uv run alembic revision --autogenerate -m "描述本次变更"

# 把未执行的迁移应用到数据库
uv run alembic upgrade head

# 查看当前数据库迁移版本
uv run alembic current

# 回退一个版本（慎用，开发期偶尔用）
uv run alembic downgrade -1
```

## 7. 遇到的问题与解决过程（重要，供回溯）

### 问题 1：`alembic revision` 报 `UnicodeDecodeError: 'gbk' codec can't decode`

- **现象**：执行 `uv run alembic revision --autogenerate` 直接崩溃在读取 `alembic.ini` 处。
- **原因**：Windows 上 Python 的 `configparser` 用系统 locale 编码（GBK）读 ini 文件，而 `alembic.ini` 里写了中文注释（UTF-8 保存），GBK 解不出来。
- **解决**：把 `alembic.ini` 里的中文注释改成英文。**经验：Windows 环境下 ini/cfg 这类配置文件不要写中文注释**（`.py` 文件不受影响，Python 源码默认 UTF-8）。

### 问题 2：`alembic revision` 报 `Target database is not up to date`

- **现象**：修复问题 1 后再次生成迁移，报目标库版本不一致。
- **原因**：此前一次被中断的执行已经生成了迁移文件 `26321e0bb705_init_business_tables.py`，但还没有 `upgrade` 到数据库，Alembic 拒绝在"有未应用迁移"时再生成新迁移。
- **解决**：检查发现已有迁移文件内容完整正确（6 张表全齐），无需重新生成，直接 `uv run alembic upgrade head` 应用即可。

### 问题 3：`setup_checkpointer.py` 报 `ModuleNotFoundError: No module named 'app'`

- **现象**：`uv run python scripts/setup_checkpointer.py` 找不到 `app` 包。
- **原因**：直接以脚本路径运行时，`sys.path` 的根是 `scripts/` 目录而不是 `backend/`。
- **解决**：脚本开头加 `sys.path.insert(0, 项目根目录)`。后续 `scripts/` 下的独立脚本都沿用此写法。

### 问题 4：PowerShell 控制台中文乱码

- **现象**：接口返回的中文在 PowerShell 控制台显示为乱码（如豆包回复、脚本 print 的中文）。
- **原因**：PowerShell 5.1 默认 GBK 编码，与 UTF-8 输出不匹配。**仅显示问题，数据本身正确**。
- **解决/结论**：验证时改用"写入 UTF-8 文件再读取"的方式确认内容正确；不影响 API 与数据库的实际行为，前端和 Swagger 中显示均正常，不做额外处理。

### 问题 5：一行式 psycopg 命令挂起 + PowerShell 引号转义踩坑

- **现象**：用 `python -c "..."` 一行式建库命令执行 110 秒无响应被中断；另一条一行式验证命令因 PowerShell 引号转义错乱报语法错误。
- **解决**：凡是涉及数据库/复杂字符串的操作一律写成 `scripts/*.py` 脚本文件执行，并给数据库连接加 `connect_timeout=5`。**经验：Windows + PowerShell 环境下避免复杂的 `python -c` 一行式命令**。
