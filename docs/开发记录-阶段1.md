## 2. 工作流拓扑（最终实现）

```
START
  ↓
topic_generator（豆包生成 5 个候选选题，落库 topics 表）
  ↓
topic_gate 【interrupt ①：waiting_topic_selection】
  ├─ 人工选定 → content_writer
  └─ 要求换一批（可带意见）→ 回到 topic_generator（batch+1，避开上批标题）
  ↓
content_writer（首次生成 / 按全部历史反馈重写，每版落库 content_versions 表）
  ↓
content_gate 【interrupt ②：waiting_content_review】
  ├─ 通过 → storyboard_planner
  └─ 驳回（必须带修改意见）→ 意见追加进 review_feedbacks → 回到 content_writer
  ↓
storyboard_planner（文案压缩成 4 张分镜：摘要 + 生图 prompt）
  ↓
image_generator（Seedream 逐张出图，串行防限流，落库 images 表）
  ↓
image_gate 【interrupt ③：waiting_image_review】
  ├─ 全部通过 → platform_adapter
  └─ 驳回（带重绘清单 [{sequence, hint}]）→ 回到 image_generator 只重绘指定张
  ↓
platform_adapter（生成小红书内容包，落库 publish_records 表，任务标记 completed）
  ↓
END
```

三个 interrupt 断点由 LangGraph `interrupt()` 实现，决策通过 `Command(resume=决策数据)` 恢复；每一步状态由 `AsyncPostgresSaver` 写入 PostgreSQL checkpoint 表，服务重启不丢。

## 4. API 一览

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/v1/tasks` | 创建任务（`{"direction": "..."}`），立即返回，后台跑工作流 |
| GET | `/api/v1/tasks` | 任务列表 |
| GET | `/api/v1/tasks/{id}` | 任务详情：`status` / `current_stage` / `running` / `pending_decision`（interrupt 载荷） |
| POST | `/api/v1/tasks/{id}/decisions/topic` | `{"action":"select","topic_id":"..."}` 或 `{"action":"regenerate","feedback":"..."}` |
| POST | `/api/v1/tasks/{id}/decisions/content` | `{"approved":true}` 或 `{"approved":false,"feedback":"..."}` |
| POST | `/api/v1/tasks/{id}/decisions/images` | `{"approved":true}` 或 `{"approved":false,"redraw":[{"sequence":2,"hint":"..."}]}` |
| GET | `/api/v1/tasks/{id}/images` | 图片列表（含 `/images/xx.png` 可直接访问的 URL） |
| GET | `/api/v1/tasks/{id}/platform-contents` | 小红书内容包（title/body/tags/images） |
| GET | `/api/debug/tasks/{id}/state` | 调试：查看 LangGraph checkpoint State |

决策接口的通用防护：任务非 running、正在执行中、无挂起断点、断点类型不匹配时分别返回 409，防止误提交。

## 5. 关键设计决策

### 5.1 工作流后台执行 + 前端轮询

LLM 链路一次要跑 30 秒~几分钟，API 里同步等待会超时。实现方式：

- `POST /tasks` 和三个决策接口只负责「触发」：`asyncio.create_task()` 把 `graph.ainvoke()` 丢到后台，立即返回 201/202；
- 图跑到下一个 `interrupt()` 或 END 自动停下，状态已在 checkpoint 里；
- 前端/脚本轮询 `GET /tasks/{id}`，通过 `current_stage` + `running` + `pending_decision` 判断进展。

### 5.2 interrupt 的「节点重执行」特性

LangGraph 的 `interrupt()` 恢复时**从节点开头重新执行**（不是从 interrupt 那一行继续）。因此三个 gate 节点在 `interrupt()` 之前只做幂等操作（更新任务阶段字段），所有落库动作放在拿到决策**之后**，避免重复写入。

### 5.3 State 设计：单线覆盖，不用 reducer

流水线是严格串行的（同一时刻只有一个节点在写 State），不存在并发合并问题，因此 `ContentTaskState` 用普通 TypedDict，节点直接整体覆盖自己负责的字段。反馈历史 `review_feedbacks` 由 content_gate 节点显式 append 后整体写回。

### 5.4 双库分工

- **checkpoint 表**（LangGraph 自管）：工作流跑到哪、State 是什么 → 负责「断点恢复」；
- **业务表**（我们自管）：topics / content_versions / images / publish_records → 负责「查询展示与留档」。
  节点内落库用独立短会话（`db_ops.py`），不与 checkpoint 事务纠缠。

### 5.5 服务重启后的孤儿任务恢复

`--reload` 热重载或手动重启会杀掉正在执行的后台协程。启动时 `recover_running_tasks()` 扫描 `status=running` 的任务：

- 挂在 interrupt 上 → 不动，继续等人工决策（靠 checkpoint 天然恢复）；
- 有 checkpoint 但没挂起（执行到一半被杀）→ `ainvoke(None)` 从最近 checkpoint 续跑；
- 完全没有 checkpoint（刚创建就被杀）→ 重新启动工作流。

## 7. 遇到的问题与解决过程（重要，供回溯）

### 问题 1：Windows 上 psycopg 异步驱动与 ProactorEventLoop 不兼容

- **背景**：`AsyncPostgresSaver` 底层用 psycopg 异步连接，而 psycopg 明确不支持 Windows 默认的 `ProactorEventLoop`（源码里直接 raise `InterfaceError`）。
- **排查**：写预检脚本 `scripts/smoke_checkpointer_async.py` 先行验证，确认 `SelectorEventLoop` 下一切正常。同时查 uvicorn 源码确认：`--reload` 模式（use_subprocess=True）下 uvicorn 自动用 `SelectorEventLoop`，所以开发期 `uv run uvicorn app.main:app --reload` 天然没问题；**直接不带 --reload 启动才会命中 Proactor**。
- **解决**：
  1. `main.py` lifespan 里检测到 Proactor 直接报错并给出正确启动方式提示；
  2. 提供 `scripts/run_server.py`，非 reload 启动时强制 SelectorEventLoop。
- **经验**：Windows 上跑 psycopg async / LangGraph AsyncPostgresSaver，事件循环必须是 Selector；技术选型前先写小脚本预检，比集成后再排查省很多时间。

### 问题 2：选题 interrupt 返回的 topic_id 全是 "None"

- **现象**：里程碑 A 自测提交选题决策时报 422：`topic_id "None" 不是合法 UUID`。
- **原因**：`save_topics()` 里 `Topic(...)` 构造后**立即**读 `row.id` 拼返回值，但 SQLAlchemy 的 `mapped_column(default=uuid.uuid4)` 是在 **flush 时**才生成主键，构造时 `row.id` 还是 `None`，于是 `str(None)` 进了 State 并被 checkpoint 固化。
- **解决**：落库前显式 `Topic(id=uuid.uuid4(), ...)` 自己生成主键。
- **经验**：ORM 对象「构造后、flush 前」默认值尚未生效；要在 commit 前拿到主键，要么显式赋值，要么先 `session.flush()`。

### 问题 3：`--reload` 监听目录过宽，测试脚本一保存就把工作流杀了

- **现象**：里程碑 A 首次自测超时——任务一直停在 `topic_generating`。查服务日志发现：保存测试脚本触发了 WatchFiles 热重载，重载杀死了正在跑选题节点的后台协程，任务成了「孤儿」。
- **原因**：`uvicorn --reload` 默认监听整个 backend 目录，`scripts/` 下写测试脚本也会触发重载；后台 `asyncio.Task` 随进程重启直接消失。
- **解决**（两层）：
  1. 启动命令改为 `--reload-dir app`，只监听业务代码目录；
  2. 实现 `recover_running_tasks()`：启动时扫描 running 状态任务，按「挂起中 / 半途被杀 / 从未开跑」三种情况分别恢复（见 5.5 节）。修复后热重载对进行中任务不再是致命的。
- **经验**：「后台 asyncio 任务 + 热重载」天然冲突，凡是这种架构都要设计重启恢复逻辑——这本来就是 checkpoint 存在的意义，顺手把里程碑 C 的能力做扎实了。

### 问题 4：PowerShell 控制台中文乱码（延续阶段 0 的问题）

- **现象**：测试脚本 print 的中文、服务日志的中文在控制台全是乱码。
- **解决**：沿用阶段 0 经验——自测脚本控制台只输出 ASCII 摘要，选题/文案/内容包等中文内容全部写入 `scripts/out/*.log`（UTF-8）再核对。数据本身（数据库、API 返回）全程正确。

### 问题 5：LLM 返回的 JSON 偶尔带 markdown 代码块包裹

- **现象**：豆包返回选题/分镜 JSON 时，有时包 ```` ```json ```` 代码块，有时前面带一句说明文字，直接 `json.loads` 会崩。
- **解决**：写了 `json_utils.parse_llm_json()`：先剥代码块，再整体解析，失败则截取首个 `[...]` / `{...}` 再解析；同时所有 prompt 都以「严格按以下 JSON 格式输出，不输出多余文字」收尾，双保险。
