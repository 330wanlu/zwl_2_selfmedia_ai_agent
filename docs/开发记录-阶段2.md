## 2. 阶段 0 / 1 / 2 如何串在一起（整合流程）

阶段不是三套独立代码，而是**同一后端上的纵向叠加**：

```
阶段 0（底座）
  FastAPI + PostgreSQL + 业务表 + 豆包/Seedream + llm_call_logs
        ↓ 被阶段 1 直接调用
阶段 1（流水线）
  LangGraph：选题 → 文案 → 分镜 → 出图 → 小红书内容包
  任务 API + 三个决策 API + checkpoint 持久化
        ↓ 被阶段 2 补齐「可调试 / 可导出 / 可标记」
阶段 2（完善）
  Debug 查 State / checkpoint 历史 / LLM 日志
  内容包 JSON 导出 + 图片 zip
  人工发布后标记 published
  Prompt 调优让产出更接近可发水准
```

运营侧（Swagger / 脚本）完整路径：

```
POST /api/v1/tasks                    创建任务（direction）
  → GET  /api/v1/tasks/{id}           轮询到 waiting_topic_selection
  → POST .../decisions/topic          选定选题
  → GET  /api/v1/tasks/{id}           轮询到 waiting_content_review
  → POST .../decisions/content        通过或带 feedback 驳回重写
  → GET  /api/v1/tasks/{id}           轮询到 waiting_image_review
  → POST .../decisions/images         通过或指定 sequence 重绘
  → GET  /api/v1/tasks/{id}/platform-contents   查看内容包
  → GET  /api/v1/tasks/{id}/export              下载 JSON
  → GET  /api/v1/tasks/{id}/export/images.zip   下载图片
  → POST /api/v1/tasks/{id}/publish/mark        标记已发布

调试旁路（DEBUG=true）：
  GET /api/debug/tasks/{id}/state
  GET /api/debug/tasks/{id}/history
  GET /api/debug/tasks/{id}/llm-calls
```

## 7. 遇到的问题与解决 / 注意点

### 问题 1：阶段 2「代码已半成品、缺验收与文档」

- **现象**：`debug` / `publish` / prompts / `test_stage2.py` 已在仓库中，但没有跑通记录和开发文档。
- **处理**：补齐自测（旧任务 API 测 + 新方向 E2E），并写本文档；本轮未再改业务逻辑，确认已实现部分符合验收标准。

### 问题 2：导出时机与 `published` 状态

- **现象**：E2E 中先 `export` 再 `mark`，导出 JSON 里 `status` 仍为 `pending`、`published_at` 为空。
- **结论**：符合设计——导出快照反映导出时刻的发布状态；标记已发布后再次导出才会带 `published`。前端阶段可约定「发布后再导出」或「导出后仍可标记」。

### 问题 3：Windows 控制台中文乱码（延续）

- **处理**：自测脚本控制台 ASCII 摘要 + UTF-8 写 `scripts/out/*.log` / `*.json`，与阶段 0/1 一致。

### 问题 4：全流程耗时与额度

- **观察**：新方向全流程约 4 分钟（无重绘）；LLM 调用 9 次（选题 1 + 文案 2 + 分镜 1 + 生图 4 + 适配 1）。
- **经验**：阶段 2 验收若只需验证导出/Debug，可复用已 completed 任务，避免重复烧额度；Prompt 质量验收再开新方向任务。
